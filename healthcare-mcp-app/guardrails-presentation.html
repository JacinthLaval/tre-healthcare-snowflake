Implementation: Where & How

Technical Detail
❄

Snowflake Grants

Layer 1
-- Read-only role for the agent CREATE ROLE ontology_reader; GRANT USAGE ON DATABASE TRE_HEALTHCARE_DB TO ROLE ontology_reader; GRANT SELECT ON ALL VIEWS IN SCHEMA TRE_HEALTHCARE_DB.ONTOLOGY TO ROLE ontology_reader; -- No INSERT, UPDATE, DELETE, DDL
Enforced by Snowflake regardless of SQL content. The agent's service role can never modify data.
⚫

SQL Validator — mcp-client.ts

Layer 2
function validateSQL(sql: string): boolean { const blocked = /\b(INSERT|UPDATE|DELETE |DROP|ALTER|CREATE|TRUNCATE |MERGE|GRANT|REVOKE|COPY |PUT|REMOVE)\b/i; if (blocked.test(sql)) { console.error('Blocked:', sql); return false; } return true; }
Runs before executeSQL() — the single chokepoint between agent and database. Rejects anything that isn't a read operation.
👁

User Confirmation

Layer 3
// Show SQL before executing const approved = await showSQLPreview(sql); if (approved && validateSQL(sql)) { const data = await client.executeSQL(sql); }
Human-in-the-loop. User sees the SQL and decides whether to run it. Ideal for sensitive datasets.
